# Vault Directory Layout Spec

**Resolves:** OQ-N (vault layout) from the PRD Open Questions list.
**Referenced by:** FR-3, FR-29, FR-30, FR-32, plus Architecture D-6, D-7, D-8.
**Implementing package:** `packages/vault-layout` (TypeScript source-of-truth) + `crates/vault-scoping` (Rust write-watcher).
**Status:** v1.0 — produced 2026-05-26 at M0 close. Re-evaluate at M5 finalization of OQ-B memory precedence.

## Why this spec exists

Every M1 story that touches the vault — Story 1.6 itself, Story 1.7 (wizard vault location), Story 1.12 (Dev persona spawn writing to its scoped dir), Story 1.13 (`claude.md` projection from the canonical persona file), Story 1.14 (per-persona log files), Story 1.15 (restart survival) — implicitly assumes the same directory layout. Same for M2 (`agy.md`), M3 (dynamic-persona vault dirs + conflict logs), M4 (BMAD workflow artifacts landing at predictable paths), and M5 (GitHub-sync filter + Supermemory category routing).

If we don't pin the layout in writing before any of those stories run, every story will invent its own convention and we'll spend M2–M5 reconciling. Pin it once, here.

## Top-level layout

The Obsidian vault is the user's local memory directory. The user picks its filesystem location in the first-run wizard (Story 1.7). Inside, the workspace owns this structure:

```
<vault>/
├── .vault-layout-version             # plaintext file containing "1.0" (this spec's version)
├── README.md                         # human-readable orientation, copy of this spec's "Quick reference"
├── personas/                         # per-persona scoped directories (D-7)
│   ├── dev/                          # fixed persona (Story 1.12)
│   ├── frontend-designer/            # fixed persona (Story 2.2)
│   └── <dynamic-persona-id>/         # spawned via BMad Builder (Story 3.3+)
└── projects/                         # project-scoped state (FR-30)
    └── <project-id>/                 # one subdirectory per Paperclip project
        ├── bmad/                     # BMAD workflow artifacts (FR-30, FR-25, M4)
        ├── personas/                 # per-project-per-persona overlays
        ├── reviews/                  # ephemeral-agent output (Story 3.6)
        ├── .project-context.md       # optional, generated by bmad-generate-project-context
        ├── .decision-log.md          # cross-persona decisions for this project
        └── .workflow-state.json      # current BMAD-workflow run state (Story 4.4)
```

The `.vault-layout-version` file at vault root lets future workspace versions detect old layouts and migrate them. M0 ships **v1.0**.

## `personas/<persona-id>/` — per-persona scope (D-7, FR-29)

Each persona owns one directory under `personas/`. The `persona-id` is a lowercase kebab-case slug — for fixed personas, `dev` and `frontend-designer`; for dynamic personas, derived from the BMAD persona ID or the BMad Builder custom slug.

```
personas/<persona-id>/
├── persona.md                        # CANONICAL persona file — see "Persona file is canonical" below
├── claude.md                         # OR agy.md OR agent.md — auto-projected from persona.md (D-6)
├── log/
│   ├── 2026-05-26.jsonl              # one file per day, JSON-lines, structured logs (Story 1.14)
│   └── …
├── skills/                           # per-persona Hermes-style skills (markdown files)
│   ├── <skill-name>.md
│   └── …
├── memory/                           # persona-scoped notes and recall content
│   └── …
├── conflict-log.md                   # last-writer-wins loser snapshots from persona-sync (D-6)
├── out-of-scope-writes.log           # JSON-lines log of writes outside scope (D-7, best-effort)
└── .persona-meta.json                # runtime metadata: lifecycle, backing CLI, model, etc.
```

### Persona file is canonical

`persona.md` is the **single source of truth** for what the persona is. CLI-specific tool configs (`claude.md` for Claude Code, `agy.md` for Antigravity CLI, `agent.md` as a generic fallback) are derived from it and live in the same directory for convenience. The `packages/persona-sync` package keeps them in sync with last-writer-wins + a per-persona conflict log (D-6).

`persona.md` follows BMAD's persona-file format (consumed by the persona's runtime activation logic, e.g., `bmad-agent-analyst/SKILL.md`'s "On Activation" sequence). For fixed personas, `persona.md` is copied at first-spawn from BMAD's library at `_bmad/bmm/agents/<persona-type>/`; for dynamic personas, `persona.md` is written by the BMB "Add Agent" panel (Story 3.10).

### `.persona-meta.json` schema (v1.0)

```jsonc
{
  "$schema": "1.0",
  "persona_id": "dev",
  "persona_type": "fixed", // "fixed" | "dynamic"
  "lifecycle": "persistent", // "persistent" | "ephemeral" (only meaningful for dynamic)
  "backing_cli": "claude-code", // see @c4n/core glossary BACKING_CLIS
  "default_model": "claude-opus-4.6-thinking",
  "spawned_at": "2026-05-26T01:42:24Z",
  "vault_dir_version": "1.0",
}
```

This file is read by `crates/persona-supervisor` and `packages/persona-sync` at spawn time. M0 stub captures the schema; M1 Story 1.12 writes the first instance for the Dev persona.

### `log/` rotation policy

- One file per UTC day, named `YYYY-MM-DD.jsonl`.
- Lines are JSON objects with at minimum `{ ts, level, message }`; additional fields are persona-specific.
- Retention defaults to 30 days; configurable per workspace.
- M2 telemetry tap (Story 2.16) writes its parsed token events to a sibling file `log/YYYY-MM-DD-telemetry.jsonl` so telemetry can be cleared without losing transcript context.

### `skills/` and `memory/`

Both are markdown-with-frontmatter directories. The distinction:

- **`skills/`** are reusable capabilities the persona can invoke (Hermes-style auto-created skills, or BMAD modular skills imported from `_bmad/`).
- **`memory/`** is persona-scoped knowledge — observations, decisions, references the persona accumulates over time. Not directly invoked; consulted at the persona's discretion.

The memory-tier precedence (Architecture D-8) reads from `memory/` first when answering a persona-scoped query.

### `conflict-log.md`

Appended-to by `packages/persona-sync` (D-6) whenever two simultaneous edits collide. Each entry:

```markdown
## 2026-05-26T14:22:11Z

**Lost edit** (persona.md): <full content of the loser file at time of conflict>

**Winning source:** claude.md (saved 2026-05-26T14:22:10Z, ~1s earlier)

**Resolution:** last-writer-wins per D-6. Review the lost edit and re-apply manually if needed.
```

### `out-of-scope-writes.log`

JSON-lines log written by `crates/vault-scoping` (D-7, Story 3.5) when a persona's process attempts to write outside its scope (`personas/<persona-id>/` or the shared project area). **Detection only**, not enforcement — per FR-29 the workspace logs but does not block (hard sandboxing is deferred to a future story, N-4).

```jsonl
{
  "ts": "2026-05-26T15:00:00Z",
  "attempted_path": "vault/personas/architect/persona.md",
  "write_type": "modify",
  "caller_persona_id": "dev"
}
```

## `projects/<project-id>/` — project-wide state (FR-30)

Each Paperclip project gets one subdirectory under `projects/`. The `project-id` is the Paperclip project's UUID (or its slug if Paperclip exposes one).

```
projects/<project-id>/
├── bmad/                             # BMAD workflow artifacts (FR-25, FR-26, M4)
│   ├── briefs/<brief-run>/brief.md
│   ├── prds/<prd-run>/prd.md
│   ├── architecture.md
│   ├── epics.md
│   ├── stories/<epic>/<story>.md      # one file per story, written by bmad-create-story (Story 3.10)
│   ├── qa/<story>/<review>.md
│   └── .decision-log.md
├── personas/                          # per-project overrides — empty by default
│   └── <persona-id>/                  # if present, overrides global personas/<persona-id>/ for this project
├── reviews/                           # ephemeral-agent output (Story 3.6)
│   └── <utc-ts>-<persona-id>-<task-slug>/
│       └── review.md
├── .project-context.md                # bmad-generate-project-context output (optional)
├── .decision-log.md                   # cross-persona project-level decisions
└── .workflow-state.json               # current BMAD workflow run state (Story 4.4)
```

### `bmad/` is BMAD-format artifacts only

The directory mirrors what `_bmad-output/planning-artifacts/` looks like for **this** project (the 4neverCompany OS meta-project), but for the user's downstream projects. The same shape so BMAD's own skills can read from either.

When a user runs `bmad-prd` inside a Paperclip project, the run folder lands at `projects/<project-id>/bmad/prds/prd-<project_name>-<date>/`.

### `projects/<project-id>/personas/<persona-id>/` — project-specific persona overlays

Optional. If present, the workspace **merges** these files over the global `personas/<persona-id>/` files using a documented rule:

- `persona.md` in the project overlay **replaces** the global one for this project's spawn.
- `skills/*.md` in the project overlay are **added** to the global set (no replacement).
- `memory/*.md` in the project overlay are scoped per-project — never merged into the global memory.

Most users won't need this. It's there for the case where a user wants Dev to behave slightly differently for one project (e.g., a particular client's coding conventions).

### `reviews/` — ephemeral-agent output

When an ephemeral persona runs (Story 3.6 — typical example: a Security Reviewer for one PR), it writes its output to `reviews/<utc-ts>-<persona-id>-<task-slug>/`. The directory is created fresh for each ephemeral spawn; the persona writes any output (typically a single `review.md` plus maybe attachments) and then exits. The directory is **preserved after exit** so the review is reviewable by the user later; only the _agent process_ is cleaned up.

### `.decision-log.md` (project-level)

Cross-persona project decisions. BMAD's per-stage decision logs (Analyst's, PM's, Architect's) link up here so a future debugger can see the full chain.

### `.workflow-state.json` schema (v1.0)

Persisted by `packages/workflow-engine` (D-12). Lets a workflow pause across desktop-app restarts (Story 4.4):

```jsonc
{
  "$schema": "1.0",
  "workflow_id": "greenfield-fullstack",
  "started_at": "2026-05-26T10:00:00Z",
  "current_phase": "architect",
  "phases_completed": ["analyst", "pm"],
  "active_personas": ["dev", "frontend-designer", "architect"],
  "last_approval_gate": {
    "phase": "pm",
    "approved_at": "2026-05-26T11:30:00Z",
    "approver": "user",
  },
}
```

## File and naming conventions across the vault

- **Directory names** are lowercase kebab-case (`frontend-designer`, not `Frontend Designer`).
- **Markdown files** use lowercase kebab-case names (`persona.md`, `conflict-log.md`).
- **Logs** use `YYYY-MM-DD.jsonl` (ISO 8601 date).
- **Timestamps** in file content are ISO 8601 UTC strings (e.g. `2026-05-26T14:22:11Z`).
- **JSON files** are pretty-printed with 2-space indent. JSON-lines logs are not pretty-printed.
- **Persona IDs** match the glossary in `packages/core/src/glossary.ts` (`FIXED_PERSONAS`, plus dynamic IDs validated against the kebab-case rule).
- **Hidden / state files** start with `.` (`.vault-layout-version`, `.persona-meta.json`, `.workflow-state.json`, `.project-context.md`, `.decision-log.md`).

## What gets GitHub-synced (M5 Story 5.4 input)

The opt-in GitHub sync (FR-33) syncs the user's vault to a personal/team repo. The default sync policy:

| Path                                           | Synced by default?       | Why                                                                           |
| ---------------------------------------------- | ------------------------ | ----------------------------------------------------------------------------- |
| `personas/*/persona.md`                        | ✓                        | Canonical persona files are config-as-code per brief §7                       |
| `personas/*/skills/*.md`                       | ✓                        | Skills are config-as-code                                                     |
| `personas/*/memory/*.md`                       | ✓ (opt-out per-category) | User-curated notes are valuable cross-machine                                 |
| `personas/*/log/*.jsonl`                       | ✗                        | Transient runtime logs; high volume; not useful cross-machine                 |
| `personas/*/conflict-log.md`                   | ✗                        | Local audit trail; doesn't transfer well                                      |
| `personas/*/out-of-scope-writes.log`           | ✗                        | Local audit trail                                                             |
| `personas/*/.persona-meta.json`                | ✓                        | Lets the other machine re-spawn the persona with matching lifecycle/CLI/model |
| `personas/*/claude.md` / `agy.md` / `agent.md` | ✗                        | Auto-derived from `persona.md`; sync the source instead                       |
| `projects/*/bmad/**`                           | ✓                        | BMAD artifacts are the project's planning record                              |
| `projects/*/personas/**`                       | ✓                        | Project-specific overlays travel with the project                             |
| `projects/*/reviews/**`                        | ✓                        | Ephemeral outputs are valuable history                                        |
| `projects/*/.project-context.md`               | ✓                        | Auto-generated context for AI consumers                                       |
| `projects/*/.decision-log.md`                  | ✓                        | The project's decision record                                                 |
| `projects/*/.workflow-state.json`              | ✗                        | Runtime state; the destination machine re-creates fresh                       |

The list above is the **default**. Each category is overridable per FR-31 (opt-in per content category for Supermemory) and the equivalent setting for GitHub sync.

## What gets Supermemory-indexed (M5 Story 5.2 input)

Even when Supermemory is enabled, the workspace only indexes categories the user opts in for. Default-OFF categories (the user must explicitly turn on):

- Decisions across projects (`projects/*/.decision-log.md`)
- BMAD artifacts (`projects/*/bmad/**`)
- Persona-scoped memory (`personas/*/memory/**`)

Default-NEVER (cannot be indexed even with opt-in — per the Privacy guardrail in PRD §11):

- Logs (`personas/*/log/**`)
- Conflict logs and out-of-scope-writes
- `.persona-meta.json` (contains references to backing-CLI credentials by name; the credentials themselves are never in vault per D-9, but metadata is still local-only)

## Migration notes

When a future workspace version changes this layout, `.vault-layout-version` lets it detect old vaults and migrate. The migration is a one-shot Story (M5 or later) and writes a `.migration-log.md` at vault root. v1.0 → v1.1 migrations should be additive (new directories, new files) rather than restructuring; restructuring requires a v2.0 bump and an explicit migration path.

## Quick reference (copy of `<vault>/README.md`)

```
<vault>/
├── personas/{persona-id}/        — per-persona scope
│   ├── persona.md                — canonical source (D-6)
│   ├── claude.md|agy.md|agent.md — auto-projected from persona.md
│   ├── log/YYYY-MM-DD.jsonl      — daily structured logs
│   ├── skills/*.md, memory/*.md  — per-persona capability + recall
│   ├── conflict-log.md           — sync-loser audit (D-6)
│   ├── out-of-scope-writes.log   — vault-scoping audit (D-7)
│   └── .persona-meta.json        — runtime metadata
└── projects/{project-id}/        — project state
    ├── bmad/                      — BMAD planning artifacts
    ├── personas/{persona-id}/     — optional per-project overlay
    ├── reviews/                   — ephemeral-agent output
    ├── .project-context.md, .decision-log.md, .workflow-state.json
    └── (other project-specific files)
```

Sync policy default: persona files + BMAD artifacts + decision logs → GitHub. Logs and audit trails → local-only.
