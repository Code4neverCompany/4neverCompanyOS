# 4neverCompany OS — Phased Build Plan

**Status:** Draft v0.1
**Companion to:** `4neverCompany_OS_Brief.md` (v0.6)
**Date:** May 2026

---

## How to Read This Document

This plan turns the brief into a sequenced build. Five milestones, each delivering something demonstrable on its own. Each milestone has a single proof goal, an explicit in-scope/deferred split, concrete deliverables, dependencies (internal and upstream), engineering surface, exit criteria, and milestone-specific risks. A pre-work section (M0) covers decisions to lock before any code is written, and a closing section covers cross-cutting concerns that run across all milestones.

Time estimates assume a focused team of 2–3 engineers. Solo work roughly doubles them. Estimates are calendar weeks, not effort weeks — they include the friction of integration with four moving upstreams (Paperclip, Hermes, BMAD, Antigravity).

**Total estimated duration:** ~24–32 weeks to a usable v1 (≈6–7 months).

---

## M0 — Pre-Work (1–2 weeks)

Before any milestone code is written, a small set of decisions and audits need to land. Skipping this phase is how bundle projects collapse three months in.

**Decisions to lock:**

- Desktop shell: Tauri vs. Electron. Decided by a one-day spike testing whether Paperclip's React UI hosts cleanly in Tauri's WebView2. If yes, Tauri. If pain is significant, Electron.
- Upstream version pinning: Paperclip vX.Y, Hermes vX.Y, BMAD vX.Y, Antigravity CLI vX.Y, Zellij vX.Y. Each pinned to a specific tag with a documented rationale and an update cadence.
- Repository structure: monorepo (pnpm workspaces, all components in one repo) vs. multi-repo (separate repos for installer, integration glue, adapters). Monorepo recommended unless team is already split across repos.
- License audit: confirm redistribution rights for every bundled component. Paperclip is MIT (clear). Hermes, BMAD, Antigravity CLI, Claude Code, and Zellij each need explicit verification of: bundling permission, redistribution rights, attribution requirements, and any commercial-use restrictions. Output: a single LICENSES.md.
- Brand/attribution policy: "powered by Paperclip / Hermes / BMAD / Antigravity" copy, where it appears, and what contributions get pushed back upstream.

**Deliverables:**

- Decision log (one-page Markdown) capturing the choices above with rationales.
- Pinned version manifest.
- LICENSES.md.
- Repo initialized with chosen structure.
- CI baseline (lint, type check, test runners — even if there's almost nothing to run yet).

**Exit criteria:** All decisions documented, version manifest in repo, license audit signed off.

## M1 — Walking Skeleton (6–8 weeks)

**Goal:** Prove the bundle can be installed and run, with one real persistent CLI agent attached to a managed terminal.

**In scope:**

The desktop shell launches and hosts the Paperclip UI. Paperclip runs locally with its embedded Postgres. Hermes Agent runs as a background process and is wired to Paperclip via the existing `hermes-paperclip-adapter`. Embedded Zellij is integrated from day one — adding it later costs more than including it now. One fixed Dev persona, backed by Claude Code, spawns into a Zellij pane on project creation. The Obsidian vault is initialized with a documented project directory layout (even though no BMAD artifacts are written yet). A minimal first-run wizard collects: model API keys (Anthropic at minimum), vault location, and Claude Code authentication. The Windows installer is generated by the chosen shell (Tauri preferred).

**Deferred:** BMAD persona library, Frontend Designer agent, message bus, dynamic spawning, Supermemory, GitHub sync, Antigravity CLI, macOS/Linux installers.

**Deliverables:**

A working Windows `.exe` installer. A first-launch experience that completes the wizard, creates a project, spawns a Dev/Claude Code instance in a visible Zellij pane, and writes a baseline `claude.md`. Documentation of the vault directory layout. A demo recording of the full flow.

**Dependencies:** M0 decisions locked. Paperclip and Hermes versions confirmed stable enough to embed. Claude Code CLI installable through standard channels.

**Engineering surface:** Desktop shell configuration and app lifecycle. Installer with bundled Node + pnpm + Python (for Hermes) + Postgres binary + Zellij binary. Process orchestration for spawning and supervising Paperclip server, Hermes process, and Zellij. Persona-file-to-`claude.md` projection. First-run wizard UI. Vault initialization logic.

**Exit criteria:** Fresh install on a clean Windows 10/11 machine → completing wizard → creating a project → having a working Dev agent in a terminal pane takes under 10 minutes. The Dev agent persists across desktop-app restarts (Zellij session survives). Vault structure is documented and stable.

**Milestone risks:** Installer size and bundling complexity — five+ runtime dependencies is a lot. Zellij on Windows historically has weaker PTY handling than Unix; spike this in week 1. Paperclip + Hermes coexisting at pinned versions might surface integration friction the upstream maintainers haven't tested.

## M2 — Second Agent + Message Bus (4–6 weeks)

**Goal:** Two agents running simultaneously, communicating through a real pub/sub bus, with the foundational progress-detection heuristic in place.

**In scope:**

A second fixed persona — Frontend Designer — backed by Antigravity CLI (`agy`). Antigravity OAuth integrated into the first-run wizard. A pub/sub message bus running on top of Paperclip's event system, exposed to all agents and to the desktop UI via WebSocket. A "channels" view in the UI showing bus traffic per project in real time. Initial progress signals wired up: file changes in the project directory, vault artifact changes. Stall detection with a configurable window (default ~5 minutes of bus activity without progress). `agy.md` auto-generated with a baseline Frontend Designer persona.

**Deferred:** Dynamic spawning, BMAD library access, workflow execution, additional progress signals (story state transitions come in M4).

**Deliverables:**

A project that spawns both Dev and Frontend Designer in their own Zellij panes. Bus messages visible in the desktop UI. Demonstrable scenarios where Hermes correctly intervenes on stalls and correctly stays out of productive multi-message exchanges.

**Dependencies:** M1 complete, specifically a solid Zellij integration. Antigravity CLI stable enough at the pinned version (it was still in public preview as of May 2026 — verify state at start of M2).

**Engineering surface:** Bus protocol design (message schema, channel topology, subscriber registration). Progress-signal subscribers (filesystem watchers for project dir, vault watchers). Stall detection algorithm and its tuning surface. UI for the bus channel view. UI for Hermes intervention prompts ("the agents have been talking for 5 minutes without changing any artifact — would you like to intervene?"). Antigravity OAuth flow in the wizard.

**Exit criteria:** Dev and Frontend Designer can coordinate on a small task end-to-end without crashing. Stall detection validated against ~10 manual test scenarios (a mix of "should intervene" and "should not intervene"). Bus messages survive Paperclip restart. Two agents idling in panes don't burn significant tokens.

**Milestone risks:** Bus protocol decisions made now have long downstream effects — over-engineer and you slow M3; under-engineer and you'll refactor in M4. Progress-detection heuristic tuning is genuinely hard and will get it wrong the first time; budget for one iteration. Antigravity CLI churn (still in preview) could force mid-milestone version updates.

## M3 — Dynamic Spawning + BMad Builder Minimal (6–8 weeks)

**Goal:** The full architectural model works — any persona can be spawned on demand, persistent or ephemeral, by either Hermes or the user.

**In scope:**

The full BMAD persona library (BMM core) installed and accessible. The featured "Add Agent" panel in the UI: persona picker, backing CLI picker, model picker, lifecycle picker (persistent vs. ephemeral). Hermes can autonomously propose spawning a persona, with a user-approval prompt. The ephemeral lifecycle works end-to-end (spawn → run task → exit cleanly, no orphan processes). Persistent dynamic agents get their own Zellij pane and vault directory, same as fixed personas. Custom persona creation via BMB: write a new persona file, save it as a reusable module. Promotion path: when an ephemeral persona has been spawned 3+ times, Hermes prompts to promote it to persistent or save as a custom module.

**Deferred:** One-click BMAD workflows (those land in M4). A marketplace or shared-modules registry. The full BMB workflow editor (M3 ships persona creation only).

**Deliverables:**

A user can click "Add Agent," pick Architect, choose persistent, and see it spawn in a new pane. Hermes proposes spawning a Security Reviewer for a sensitive code change and the user approves. A custom "Data Analyst" persona can be created via BMB and used immediately.

**Dependencies:** M2 complete (bus must work — dynamic agents need to join it). BMAD installable via `npx bmad-method install` at the pinned version.

**Engineering surface:** BMB panel UI. Agent lifecycle management (spawn, dismiss, promote, persist across restarts). Persona-file-to-tool-config projection for multiple CLIs (`claude.md`, `agy.md`, generic `agent.md`). Resource limits per agent. Cost tracking per persona.

**Exit criteria:** At least five distinct persona types can be spawned successfully (Analyst, PM, Architect, SM, QA at minimum). Ephemeral agents reliably clean up — no orphan processes after 100 spawn/exit cycles. Persistent dynamic agents survive app restart same as fixed ones. Custom personas can be created and reused across projects.

**Milestone risks:** Lifecycle management bugs (orphans, zombies, resource leaks) are the failure mode here — these need stress testing, not just happy-path testing. Concurrent model API quota interactions when many agents spawn at once. BMB UX complexity creep — discipline needed to keep the "Add Agent" flow short.

## M4 — Full BMAD Workflow (4–6 weeks)

**Goal:** One-click `greenfield-fullstack` end-to-end. The day-1 UX actually works.

**In scope:**

A workflow execution engine that reads BMAD YAML workflows and runs phases sequentially with user-approval gates at each handoff. A "Start a BMAD project" entry point on the main UI. Workflow visualization: the user can see the current phase, which personas are involved, and which artifacts have been produced. Standardized BMAD artifact layout in the vault — a documented directory structure for PRDs, architecture docs, story files, and QA reports. Story-file state machine (`open → in-progress → review → done`) tied to progress signals on the bus. Story-state transitions become new progress signals (closing M2's heuristic). `brownfield` supported as a second workflow, mainly to prove the engine isn't single-workflow.

**Deferred:** Custom workflow authoring via BMB (M5+ or post-v1). Workflow forking or branching mid-run.

**Deliverables:**

A demonstrable greenfield-fullstack run from a fresh project, producing a working code skeleton plus all BMAD artifacts (brief, PRD, architecture, story files, QA reports), in under one work session. A brownfield run that ingests an existing test repo and produces a refactor plan.

**Dependencies:** M3 complete (dynamic spawning required to bring in PM, Architect, SM, QA on demand at the right phases).

**Engineering surface:** Workflow engine (YAML interpreter + phase orchestrator). Workflow visualization UI. Vault layout standardization and writers per artifact type. Story state machine and state-change events on the bus. Approval-gate UI.

**Exit criteria:** A clean greenfield-fullstack run produces a working code skeleton plus complete BMAD artifacts in one session, on at least three different test project ideas. Brownfield runs on at least one real test repo. Workflow can be paused, app closed, reopened later, and resumed cleanly.

**Milestone risks:** BMAD workflow YAML format evolves upstream — pin the version explicitly and document the migration story for future BMAD releases. "Working code skeleton" is fuzzy as an exit criterion — define one concrete test scenario at the start of the milestone. User approval gates can become an annoyance fast — design them so the default path is "approve and continue" with one keystroke.

## M5 — Memory, Collaboration, and Polish (4–6 weeks)

**Goal:** Project survives across machines and sessions; cross-project memory works; installer is hardened; macOS and Linux supported.

**In scope:**

Supermemory integration as opt-in, with a clear local-only toggle per content category. GitHub sync for configs, BMAD artifacts, skills, and personas — with an explicit policy for what gets committed vs. stays local. The full multi-machine workflow: create a project on machine A, push to GitHub, clone on machine B, install the workspace, and resume work with the same fixed personas and the same vault state. macOS installer (`.dmg`). Linux installer (AppImage or distro packages). Installer hardening: error recovery, partial-install resumption, in-place upgrades. Per-persona budget gates with alerts and automatic pause when crossed. A project-level kill switch ("stop everything now"). Basic documentation site (install, first-run, common tasks).

**Deferred:** Mobile clients, real-time multi-user collaboration, hosted SaaS — all deferred to post-v1 per the brief.

**Deliverables:**

Three working installers (Windows, macOS, Linux). A demonstrable cross-machine continuity test. A basic docs site at a stable URL. Public-beta-ready build.

**Dependencies:** M4 complete (the system needs to be functionally whole before polishing).

**Engineering surface:** Supermemory API integration and sync engine. GitHub sync logic. Cross-platform installer engineering (significant effort — each platform has its own quirks). Budget and cost tracking UI. Docs infrastructure.

**Exit criteria:** All three installers tested clean-install on at least one machine each. Round-trip test: project created on Windows, pushed to GitHub, opened and resumed on macOS, then on Linux, with full continuity. Supermemory measurably improves Hermes responses on at least one defined cross-project retrieval scenario.

**Milestone risks:** Cross-platform installer work always takes longer than expected — budget conservatively. Supermemory pricing and rate limits at real usage levels need to be tested before public beta. GitHub sync policy decisions (what's in the repo) will create friction with users who disagree; documenting the rationale clearly matters.

## Cross-Cutting Concerns

Some workstreams run across every milestone rather than belonging to one. They need owners and rhythm from M1 onward.

**Testing strategy.** Each milestone needs a defined set of scenario tests (not just unit tests) that prove the milestone-level proof goal. The full end-to-end test set grows over time; by M5 the round-trip cross-machine scenario is the headline test.

**Documentation.** User-facing docs lag features by one milestone (M2 docs land during M3, etc.) so they reflect what shipped, not what was planned. The exception is the install flow, which needs docs at M1 because that's the first thing any user sees.

**Security review.** A formal security review should happen between M3 and M4 — late enough that the architectural surface is real, early enough that findings can land in M4/M5. Areas of focus: agent sandboxing, OAuth token storage, message bus authorization, vault file permissions, Supermemory data handling.

**Cost monitoring.** Token-cost telemetry should be wired in by M2 (when the second agent lands and multi-agent costs become a real risk). Per-persona quotas land in M5 but the telemetry needs to exist earlier.

**Upstream sync rhythm.** Paperclip, Hermes, BMAD, and Antigravity all move fast. Every milestone should reserve ~10% of its capacity for rebasing on upstream updates. Skipping this turns into a forced large rebase later, which is more expensive.

**Contribution-back policy.** Any adapter, plugin, persona, or BMAD module that's general-purpose should be offered upstream. This is both good open-source citizenship and a way to reduce the long-term maintenance burden of carrying patches in-tree.

## Risk Register Summary (Plan-Level)

The four risks that span the whole plan, ranked by potential schedule impact:

1. **Upstream churn across four moving projects.** Paperclip, Hermes, BMAD, Antigravity — any one of them landing a breaking change forces work. Mitigation: tight version pinning, scheduled upstream-sync time, contribution-back to reduce divergence.
2. **Installer bundling complexity.** Five+ runtime dependencies plus three target platforms is the largest source of "engineering tax" in the plan. Mitigation: invest in installer infrastructure early (M1), don't defer to M5 polish.
3. **Process orchestration edge cases.** Zellij + multiple PTYs + dynamic spawn/exit + restart recovery is genuinely hard to get right. Mitigation: dedicated stress testing in M3 exit criteria.
4. **Cost runaway with persistent multi-agent workflows.** Even with progress-based stall detection, the bus is liberal by design. Mitigation: telemetry from M2, soft quotas before M5 hard quotas.

## What This Plan Doesn't Cover

A few things this plan deliberately leaves to later or to other documents:

- The first-run wizard mini-spec — referenced in M1 but not detailed. Worth a one-pager once M0 decisions are locked.
- The progress-signal taxonomy — referenced in M2 and M4 but not catalogued. Worth its own design doc inside M2.
- The bus protocol schema — referenced in M2 but not specified. Should be one of the first M2 deliverables, before any code.
- The vault directory layout — referenced in M1 and M4 but not laid out. Should be drafted in M1 since it shapes everything downstream.
- The marketing/positioning/website work — out of scope for this plan; track separately.

These are natural follow-up artifacts. Each is small enough (one-pager or two-pager) to live alongside the brief and this plan as the engineering work begins.
