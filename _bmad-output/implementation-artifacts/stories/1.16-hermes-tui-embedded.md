# Story 1.16 — Hermes TUI embedded as a pane

## User story

As a **solo dev wanting to observe Hermes' decisions and direct it**,
I want **the Hermes TUI embedded in the multi-terminal view as a Zellij pane**,
So that **I can scroll/inspect Hermes in the same UI surface as the agents**.

## Acceptance criteria (verbatim from epics.md)

> **Given** Story 1.11 (Zellij adapter)
>
> **When** the workspace starts a project
>
> **Then** a Zellij pane labeled "Hermes" runs Hermes' TUI binary
>
> **And** the pane behaves identically to running `hermes` standalone (no commands missing)
>
> **And** Hermes' TUI is visible alongside the Dev pane in the desktop UI

## The architectural ambiguity

"Hermes' TUI is visible … in the desktop UI" admits **three viable interpretations**. The brief locks WHAT (a fixed sibling alongside Dev) but not HOW (where the pixels live). Each interpretation pulls in a different infrastructure investment.

### Option A — Separate native Zellij window (the "1.16-lite" path)

Zellij owns Hermes in a multi-pane layout (Hermes + Dev side-by-side). Our Tauri webview shows a "Memory" panel with a single CTA: "Focus Zellij window." Clicking switches OS focus to the Zellij terminal.

**Pros:**

- Zero new dependencies. We already have everything: spawn via `c4n-zellij-adapter`, log via `c4n-persona-supervisor`.
- Preserves restart survival (the five-link chain from Story 1.15) verbatim.
- True-TUI fidelity automatically (Zellij owns the PTY).

**Cons:**

- Doesn't deliver "visible … in the desktop UI" in the natural reading — the desktop is a launcher for Zellij, not a host.
- Breaks the brand promise of "one product surface" — users alt-tab between two windows.
- Hard sell for a v1 demo.

### Option B — xterm.js embedded in the Tauri webview (the "1.16-proper" path)

A new `MemoryView.tsx` hosts xterm.js. A new Rust Tauri command spawns Hermes through `portable-pty` (real PTY, not raw stdio). PTY output streams to the frontend via Tauri events; xterm.js renders. Keyboard input from xterm.js streams back to PTY stdin.

**Pros:**

- True embedded TUI — full color, cursor positioning, keyboard input, scrolling, the works.
- Matches the natural reading of the AC: Hermes is rendered _inside_ our window, alongside Dev.
- Reusable infrastructure: the same xterm.js + PTY bridge can later host the Dev persona too (closes the Story 1.12b "embedded terminal" gap we deferred).

**Cons:**

- **Breaks Story 1.15's restart survival** if naively done: a PTY owned by our Tauri process dies when we exit. Mitigations: (a) launch the PTY via Zellij so Zellij still owns it and we just connect to its output (no clean Zellij API for this — Zellij would need a plugin), or (b) launch the PTY directly in our Rust side and accept that Hermes restarts each time the desktop opens (acceptable for v1: Hermes is stateless enough that restart cost is low; Story 1.15's restart-survival explicitly covers the Dev persona, not Hermes).
- New dependency: `xterm` + `@xterm/addon-fit` (npm) + `portable-pty` (cargo). Mature, well-maintained, but new surface area.
- Substantial code: ~300 LOC frontend (MemoryView + xterm.js wiring + event handlers) + ~250 LOC Rust (PTY spawn + read/write loops + Tauri event emission).

### Option C — Text-mirror via the supervisor (the "1.16-hybrid" path)

Keep Zellij as the spawning authority (restart survival preserved). The supervisor (already in place from Story 1.14) writes JSONL lines to disk. The frontend `MemoryView.tsx` _tails the JSONL file_ + renders the lines into a scroll buffer (NOT xterm.js — just a plain log viewer).

**Pros:**

- Reuses Story 1.14 infrastructure. Almost zero new dependencies.
- Preserves restart survival fully.

**Cons:**

- **Doesn't satisfy "behaves identically to running `hermes` standalone"** — no cursor positioning, no in-place updates, no interactive input. Hermes' TUI uses `curses`-style screen redraws; JSONL captures line-by-line output. A Hermes screen that says `[ progress: ████████░░  82% ]` becomes ten different lines instead of one updating display.
- The AC's "no commands missing" clause is impossible without interactive input — we'd have a read-only Hermes view.
- Sells short the product's premise of "agent observability."

## Restart-survival impact

Option B as written introduces a regression: Hermes is owned by our Tauri process, not Zellij, so closing the desktop kills Hermes. This is **acceptable for v1** because:

1. **Story 1.15's restart survival explicitly covers the Dev persona**, not Hermes. The AC for 1.15 mentions "Dev persona's Claude Code session."
2. **Hermes is stateless enough that re-spawn is cheap.** Unlike Claude Code (which has expensive context loading), Hermes' job is to observe + propose; its state is the vault, not in-process memory.
3. **A future story can move Hermes' PTY to be owned by a Rust sidecar daemon** (not the Tauri process) to recover restart survival. Flag as a separate story under M2+.

If you want Hermes restart survival in v1, Option A is the only zero-cost path; Option B requires the sidecar-daemon work above before shipping.

## Risks

- **portable-pty Windows behavior.** On Windows, portable-pty uses ConPTY (Windows 10 1809+). ConPTY has historically been quirky with TUI applications. Mitigation: spike `portable-pty` against `htop`-style apps before committing to a real implementation. (Add to 1.16b as a first sub-task.)
- **xterm.js bundle size.** xterm.js + addon-fit is ~250 KB minified. Acceptable but worth confirming the Vite build doesn't balloon.
- **Hermes installation path.** Hermes is Python (per Architecture). User needs `hermes` on PATH OR we need to discover the installed location. For 1.16b: env override + PATH lookup, mirroring the supervisor binary lookup.
- **Hermes restart survival regression** (already discussed above).
