# Story 1.16 — Hermes TUI embedded as a pane

## User story (verbatim from epics.md)

As a **solo dev wanting to observe Hermes' decisions and direct it**,
I want **the Hermes TUI embedded in the multi-terminal view as a Zellij pane**,
So that **I can scroll/inspect Hermes in the same UI surface as the agents**.

## Acceptance criteria (verbatim from epics.md)

> **Given** Story 1.11 (Zellij adapter)
>
> **When** the workspace starts a project
>
> **Then** a Zellij pane labeled "Hermes" runs Hermes' TUI binary
>
> **And** the pane behaves identically to running `hermes` standalone (no commands missing)
>
> **And** Hermes' TUI is visible alongside the Dev pane in the desktop UI

## The architecture (now correctly grounded in the brief)

The brief locks the shape: **Zellij-powered multi-terminal view, embedded in the Tauri webview, persistent across restarts.** That means:

1. **Zellij stays as the spawn authority for every persistent persona** (Dev today; Hermes adds in this story; Frontend Designer adds in M2; dynamic personas in M3+).
2. **Restart survival comes from Zellij** (Story 1.15's five-link chain holds across this story).
3. **The xterm.js layer in our webview is the DISPLAY** of what Zellij is running, NOT a replacement for it.
4. **The bridge from Zellij-owned PTY to xterm.js is the persona-supervisor**, upgraded from raw stdio to a real PTY. The supervisor writes raw PTY bytes to a tap file in the vault; xterm.js tails it.

The complete picture:

```
4nCO desktop window (Tauri webview)
├── ProjectsView (existing from Story 1.12)
│   └── Dev persona terminal (xterm.js, NEW in 1.16)
│       └── tails <vault>/personas/dev/log/<date>.pty.raw
│
└── MemoryView (NEW in 1.16 — was placeholder)
    └── Hermes terminal (xterm.js)
        └── tails <vault>/personas/hermes/log/<date>.pty.raw

Zellij session(s) — outlives the desktop, owns the processes
├── session: dev-<project-id>      (existing from Story 1.12)
│   └── pane: c4n-persona-supervisor + claude
│       ├── PTY bytes → its own stdout (so `zellij attach` still works)
│       ├── PTY bytes → <vault>/personas/dev/log/<date>.pty.raw  (xterm.js tap)
│       └── line capture → <vault>/personas/dev/log/<date>.jsonl (existing 1.14 telemetry)
│
└── session: hermes-<project-id>   (NEW in 1.16)
    └── pane: c4n-persona-supervisor + hermes
        └── (same three taps)
```

**Why this is architecturally honest:**

- "Zellij-powered" per the brief: ✅ Zellij still spawns, supervises, persists.
- "Embedded in the desktop UI" per the brief: ✅ xterm.js renders inside the Tauri webview.
- "Survives desktop-app restart" per the brief + Story 1.15: ✅ Zellij's session outlives the desktop; xterm.js just re-attaches its tail to the existing tap file on relaunch.
- Memory is a separate concern (D-8 four-tier): ✅ the vault saves CONTENT (artifacts, persona definitions, conversation transcripts via JSONL); Zellij saves PROCESS STATE (the running claude / hermes binaries). Both layers persist; they answer different questions.

**Why my earlier "Option B as written" was wrong:**

I had the supervisor + xterm.js doing the spawn directly (via `portable-pty` owned by our Tauri process), which would have killed the child when the desktop closed. The correct design keeps Zellij as the parent of the supervisor, with the supervisor only producing the OUTPUT TAP — the process tree stays parented to the Zellij server, not to our app.

## Multi-pane layout caveat

The full polished multi-pane view (side-by-side or tabbed, ≥ 10 panes per session, route stdin/stdout/stderr per pane, integration tests with 5+ panes) is **Story 2.4 in M2** per epics.md. Story 1.16 just adds Hermes as the second persistent persona; the multi-pane UX polish lives with Story 2.4.

For M1 Story 1.16, "visible alongside the Dev pane in the desktop UI" is satisfied by:

- Both Dev (in ProjectsView) and Hermes (in MemoryView) are simultaneously running
- Both are embedded in our Tauri webview via xterm.js
- The user navigates between them via the side rail
- A future M2 Story 2.4 lands the side-by-side multi-pane view in a single screen

This is a defensible reading of "alongside" — both are visible IN THE SAME APP, switchable in ≤ 1 click. The strict side-by-side-on-one-screen reading is Story 2.4's job.

## Sub-story breakdown

### 1.16a — Persona-supervisor PTY upgrade (foundational; required first)

The supervisor currently spawns its child with raw `Stdio::piped()` (Story 1.14a/b). Real TUI fidelity (cursor positioning, color, keyboard input) requires a PTY. Add `portable-pty` dep + refactor `supervise()` to use it. The supervisor:

- PTY-spawns its child
- Writes raw PTY bytes to `<vault>/personas/<id>/log/<date>.pty.raw` (new — the tap file)
- Continues writing line-stripped JSONL to `<vault>/personas/<id>/log/<date>.jsonl` (existing 1.14a)
- Forwards bytes to its own stdout (so Zellij's pane display is unchanged)
- Accepts input bytes from a control channel (new — file watch on `<vault>/personas/<id>/log/<date>.pty.in`, OR Unix domain socket / Windows named pipe). For 1.16a v1 the simplest is file-watching — write a chunk to `.pty.in`, supervisor reads it and forwards to PTY stdin.

**Files:** `crates/persona-supervisor/Cargo.toml` (+ portable-pty 0.9), `crates/persona-supervisor/src/lib.rs` (refactor `supervise()` + `capture_pipe()`).

**Test plan:** existing tests still pass after refactor; new test that PTY-spawns `cargo --version` and verifies the `.pty.raw` file contains the version string (in bytes, not just JSONL).

**Scope:** ~120 LOC refactor + ~40 LOC test. Doesn't ship UI by itself; foundational for 1.16b/c.

### 1.16b — Hermes spawn (mirror the Dev pattern)

New Tauri commands in `apps/desktop/src-tauri/src/commands/mod.rs`:

- `spawn_hermes(project_id)` — mirrors `spawn_dev_persona`. Session name `hermes-<project-id>`. Runs through the supervisor (now PTY-wrapped after 1.16a). Command is `hermes` (binary discovery: PATH lookup + `C4N_HERMES_BIN` env override).
- `hermes_status(project_id)` — mirrors `dev_persona_status`. Returns `HermesStatus { state: "running" | "not-running" | "missing", session_name }`.
- `kill_hermes(project_id)` — cleanup. Calls `zellij delete-session`.

**Files:** `apps/desktop/src-tauri/src/commands/mod.rs` (~150 LOC), `apps/desktop/src-tauri/src/lib.rs` (+ register).

**Test plan:** 3 unit tests mirroring Story 1.12's pattern (session name format, env-override lookup, fallback default).

**Scope:** ~150 LOC + ~50 LOC tests.

### 1.16c — Frontend xterm.js + tail bridge

New TS infrastructure:

- Add `@xterm/xterm` + `@xterm/addon-fit` + `@xterm/addon-web-links` to `apps/desktop/package.json`.
- New `apps/desktop/src/lib/PtyTail.ts` — class that wraps an xterm.js instance, opens a Tauri channel that tails `<vault>/personas/<id>/log/<date>.pty.raw`, feeds bytes into xterm.js, and writes user input back via the supervisor's `.pty.in` file.
- New `apps/desktop/src/views/MemoryView.tsx` — replaces the "Coming in Story 1.16" placeholder. Hosts a `PtyTail` instance bound to the active Hermes session. Shows status + spawn CTA when Hermes isn't running.
- Update `apps/desktop/src/views/ProjectsView.tsx` — wire a `PtyTail` into the Dev persona "Running" state, replacing the static "session_name" panel with a live terminal.

**New Rust Tauri command:** `tail_persona_pty(vault_path, persona_id)` returns a Tauri Channel/EventEmitter that streams bytes from the tap file. (Use Tauri 2's Channel API for backpressure-safe streaming.)

**Files:** `apps/desktop/package.json` (+ 3 deps), `apps/desktop/src/lib/PtyTail.ts` (~150 LOC), `apps/desktop/src/views/MemoryView.tsx` (~120 LOC), edits to `ProjectsView.tsx` (~30 LOC), `apps/desktop/src-tauri/src/commands/mod.rs` (+ tail command, ~80 LOC).

**Test plan:** mostly UI smoke + integration. The PTY-tail glue is hard to unit-test cleanly; smoke via `pnpm dev:desktop` is the realistic verification.

**Scope:** ~380 LOC across frontend + Rust. The largest sub-story.

## Risks

- **portable-pty Windows behavior on ConPTY** — known to have quirks with some TUI applications. Mitigate with a spike on `htop`-equivalent before committing real implementation.
- **Tail-file race conditions** — the supervisor writes `.pty.raw` while xterm.js reads. Use append-only writes + the OS's file-watch debounce. Standard pattern; risk low.
- **Hermes binary discovery** — Hermes is Python (per architecture); user needs Hermes installable on PATH. Wizard hasn't done this yet. Mitigation: env override `C4N_HERMES_BIN`; flag "wizard installs Hermes" as a future story.
- **`.pty.raw` file size** — for a long-running TUI session, the tap file grows unbounded. Mitigate: per-day rotation matching the existing JSONL convention; xterm.js renders only the visible window + scrollback buffer; the tap file is the persistent record.
