# Story 1.18 — End-to-end scenario test (install → wizard → project → working Dev agent ≤ 10 min)

## User story (verbatim from epics.md)

As **Maurice validating the M1 exit criterion**,
I want **a manual scenario test that proves a fresh Windows install → wizard → project → working Dev agent in a pane takes under 10 minutes**,
So that **the SM-1 success metric is demonstrable**.

## Acceptance criteria (verbatim from epics.md)

> **Given** the M1 `.exe` from Story 1.17 and a clean Windows 10 or 11 VM with no prior install
>
> **When** Maurice (or a tester) runs through the full install → wizard → project flow with a stopwatch
>
> **Then** total elapsed time from "double-click .exe" to "typing into a working Dev agent's Zellij pane" is **≤ 10 minutes**
>
> **And** a screen recording of the run is captured in `tests/manual/m1-exit-criterion-recording.mp4` (or similar)
>
> **And** any step that took longer than expected is logged in `tests/manual/m1-exit-criterion-notes.md` with a remediation plan

## What this story IS (and what it is NOT)

**IS:**
- A written manual-verification protocol at `docs/e2e-smoke-test.md` with step-by-step instructions, per-step expected timestamps, expected UI state, and per-step diagnostic fall-throughs ("if step N fails, check X first").
- An `#[ignore]`d Rust test in `apps/desktop/src-tauri/src/commands/mod.rs::tests` that surfaces the protocol via `cargo test -- --ignored` (mirror of `restart_survival_manual_verification`, Story 1.15).
- A `tests/manual/` directory scaffolded with the recording + notes filename slots the AC names, plus a README so a tester knows where to drop the captures.
- A sprint-status flip + (after Maurice runs the protocol on real hardware) a captured recording + notes commit.

**IS NOT:**
- An automated UI test. Tauri 2 + xterm.js end-to-end automation (WebDriver-style) is real work — explicit non-goal per the AC, which calls for "manual scenario test."
- A code-signing prerequisite. Alpha-grade SmartScreen warning is documented inline, not gated on.
- A blocker for Story 1.17b. The protocol explicitly accommodates the supervisor-on-PATH workaround as a one-time prerequisite outside the 10-min budget; when 1.17b ships, the protocol gets a minor edit to delete that prerequisite step.

## Failure modes (top 5)
1. **Wizard Anthropic-key validation hangs > 5s:** likely captive-portal network; check `curl https://api.anthropic.com/v1/models`.
2. **"Spawn Dev →" returns "zellij not on PATH":** the `winget install` step was skipped or the shell hasn't refreshed PATH; restart the desktop app.
3. **"Spawn Dev →" returns "c4n-persona-supervisor not found":** the `cargo install` prerequisite was skipped. Story 1.17b removes this prerequisite when it lands.
4. **Terminal embed appears blank > 5s:** the `.pty.raw` tail hasn't started; check `~/.4nevercompanyos-vault/personas/dev/log/<date>.pty.raw` exists and has bytes.
5. **Claude prompt visible but typing produces nothing:** Story 1.16d's `.pty.in` writer path. Check `<vault>/personas/dev/log/current.pty.in` is being appended to on keypress (Sysinternals Process Monitor on that file).

## Risks

- **Protocol drift.** The protocol references file paths + UI labels that come from Stories 1.7–1.16. If a future story renames a wizard step or moves a vault subpath, the protocol decays silently until someone runs it. Mitigation: when a future story changes one of these surfaces, the dev-story spec MUST include "update `docs/e2e-smoke-test.md`" as a checklist item. (Add to `references/reference-bmad-method-workflow.md` as a discipline note.)
- **Budget is an estimate, not measured.** The 10-min budget is decomposed into per-step estimates; the actual capture-run is the calibration. The first real run may show one step is wildly over budget (e.g., NSIS auto-WebView2-install on slow connections). That's surfaced via the `tests/manual/m1-exit-criterion-notes.md` capture, becomes a discovered story.
- **SmartScreen + WebView2 bootstrap are network-dependent.** Both step 1 (SmartScreen "Run anyway" detour) and step 2 (WebView2 auto-bootstrap if missing) can spike on slow / locked-down networks. Out of scope for 1.18 to mitigate; in scope to document.
- **`tests/manual/` outside the existing test layout.** Most tests live under `apps/*/src-tauri/src/`, `apps/*/src/`, or `crates/*/src/` next to their code. `tests/manual/` is a NEW top-level directory for human-run artifacts. Justification: the recording + notes are run-time captures of a manual protocol, not source. Putting them under any `src/` tree pollutes the build graph. README in that dir explains the convention.

## Code-review expectations (3-pass)

The 1.18 diff is small (one doc, one `#[ignore]`d test, one README, one `.gitkeep`, two sprint-status / story-file edits). The 3-pass should still happen, scoped tight:

- **Edge Case Hunter:** what about a tester on Windows 10 (not 11)? What about a tester whose Anthropic key is rate-limited mid-validate? What about a tester whose `winget` is stale and installs an older Zellij? The protocol should call these out under "Failure modes."

Does the test stub mention the recording + notes filenames? Does `tests/manual/README.md` explicitly call out those two filenames as the capture slots?
