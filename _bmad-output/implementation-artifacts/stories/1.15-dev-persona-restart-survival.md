# Story 1.15 — Dev persona survives desktop-app restart

## User story

As a **solo developer who opens 4neverCompany OS, starts working with the Dev persona, then closes the desktop app for the day**,
I want **the Dev persona's Claude Code session to still be there when I reopen the project the next morning**,
So that **I don't lose conversation context or have to re-spawn / re-auth between dev sessions**.

## Acceptance criteria (verbatim from epics.md)

> **Given** a project is open with the Dev persona spawned (Story 1.12)
>
> **When** the desktop app restarts
>
> **Then** the user reopens the project and the same Zellij session is re-attached

## Restated AC with sub-conditions

1. **Active-project pointer survives a desktop restart.** `~/.4nevercompanyos/active-project.toml` (Story 1.12a) is on the user's filesystem, not in-memory; reopening reads it back.
2. **Zellij session survives a desktop restart.** This is the D-2 architecture guarantee: Zellij is a separate server process that outlives the desktop app. Per `crates/zellij-adapter/src/lib.rs` rustdoc: "the Zellij server outlives the desktop app's lifetime; closing the workspace doesn't kill the persona's CLI process."
3. **The supervisor + claude inside the Zellij session survive a desktop restart.** Story 1.14b: `c4n-persona-supervisor` is the immediate child of the Zellij pane, `claude` is its child. Zellij owns the PTY; the desktop never had handles on these processes. Killing the desktop has no effect.
4. **Reopening the project does NOT spawn a duplicate.** When the desktop comes back and `spawn_dev_persona` runs (either auto or via a UI click), the existing-session-reuse branch returns `DevPersonaStatus::Running { session_name }` without re-spawning.
5. **The UI reflects the running state on launch.** ProjectsView's `dev_persona_status` 3s poll picks up the running session within the first poll cycle (<3s) and renders the "attached" badge.

## Test plan

### Automated

- `cargo test -p c4n-desktop`: existing 11 tests still pass.
- New ignored test must compile under `cargo test --no-run`.

### Manual (the protocol the ignored test documents)

Run on a real Windows 11 dev box with Zellij ≥ 0.44.3 + Claude Code + `c4n-persona-supervisor` on PATH:

1. `pnpm dev:desktop` — launch the desktop app.
2. Open a project: Projects panel → "Open project →" → pick a folder.
3. Spawn the Dev persona: "Spawn Dev persona →" → wait for green "attached" badge.
4. Verify externally: `zellij list-sessions` shows `dev-<project-id>`; `zellij attach dev-<project-id>` shows the Claude Code prompt.
5. Detach (`Ctrl+Q` inside Zellij, or just don't attach).
6. **Close the desktop app entirely.** Verify no `c4n-desktop` process remains (Task Manager / `Get-Process c4n-desktop`).
7. Verify Zellij side: `zellij list-sessions` still shows `dev-<project-id>`. `zellij attach` still shows the Claude Code prompt — conversation history intact.
8. Detach again.
9. `pnpm dev:desktop` — relaunch the desktop app.
10. **Expected:** ProjectsView loads showing the prior project's name, path, and "attached" badge within ~3 seconds (first poll cycle).
11. **No spawn re-runs.** `zellij list-sessions` still shows exactly one `dev-<project-id>` session — not a duplicate.
12. Click "Refresh" to force a fresh `dev_persona_status` call; badge stays green.

### Pass condition

Steps 7, 10, and 11 all hold. If 7 fails, Zellij or the supervisor is unstable on this platform (out of 1.15's scope — flag a follow-up). If 10 fails, the active-project pointer didn't survive (regression in Story 1.12a). If 11 produces a duplicate, the session-reuse branch in `spawn_dev_persona` regressed.

## Risks / open items

- **Zellij version drift.** v0.44.0 added native Windows ConPTY support; v0.44.3 has additional mouse-event fixes. Older Zellij ≤ 0.43 won't restart-survive on Windows. The pinned version `0.44.3` is in `docs/pinned-versions.md`; restart survival is contingent on the user actually having that version.
- **Stale active-project pointer.** If the user deletes the project folder while the desktop is closed, reopening will see `current_project()` return a path that no longer exists. The current implementation surfaces this as a generic error. A future story could detect-and-offer-to-clear; not in scope for 1.15.
- **Multiple desktops.** The architecture assumes single-instance desktop. Running two desktop instances against the same vault would race on `active-project.toml`. Out of 1.15's scope — flag as M2+ if multi-window becomes a real requirement.
