# Story 2.5 — Both fixed personas survive desktop-app restart

## User story

As a **solo dev who closes and reopens the app**,
I want **both Dev and Frontend Designer to reattach to their existing panes**,
So that **neither is lost on restart**.

## Acceptance criteria (verbatim)

> **Given** Story 1.15 (Dev restart survival) extended for Frontend Designer
>
> **When** I close the desktop app with both fixed personas running and reopen it
>
> **Then** both panes reattach to their existing Claude Code and `agy` processes; zero orphans
>
> **And** both pane scrollbacks show >= the last 1000 lines

## Implementation scope

**This story requires NO new runtime behavior.** The Frontend Designer's restart survival is already provided by the same five-link chain documented for Story 1.15 in `docs/restart-survival.md`. `spawn_designer_persona` (added in Story 2.2) already has:

- the session-reuse branch (returns `Running` if `designer-<id>` exists — no duplicate spawn),
- `close_on_exit: false` (pane survives even if `agy` exits),

and `PersonasView.tsx` (Story 2.3) already polls `designer_persona_status` every 3s, so the pane re-attaches in the UI on launch. Zellij's daemon ownership keeps the `agy` process (and its pane scrollback) alive across a desktop restart exactly as it does for `claude`.

Like 1.15, this story closes by adding the verification artifacts — the designer spawn path previously had **zero** test coverage.

## The `agy`-vs-Claude-Code session-persistence concern

The story's Technical Context flags: "`agy` session persistence vs Claude Code session persistence differ." This is a non-issue for OS-level restart survival: restart survival **never restarts either CLI**. Zellij owns the PTY and parents the supervisor + CLI; closing the desktop just drops a client. Each process keeps running with its in-memory state intact, so neither CLI's own session-resume is exercised — any difference between them is irrelevant. See `docs/restart-survival.md` § "The `agy`-vs-Claude-Code session-persistence concern".

## Scrollback ≥ 1000 lines

Carried by Zellij's per-pane scrollback buffer (held in the Zellij server's memory), which survives because the pane survives. Zellij's `scroll_buffer_size` default is 10000 — well above the 1000-line floor — so the AC holds for both panes out of the box.
