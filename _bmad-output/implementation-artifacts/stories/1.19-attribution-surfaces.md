## What this story IS (and what it is NOT)

**IS:**

- Three consumer surfaces that pull from the existing `packages/core/src/attribution.ts`:
  1. **Wizard `DoneStep.tsx`** — replace the existing hardcoded one-line placeholder with a real "Powered by…" block (component name, license, link to LICENSES.md) sourced from `attribution.ts`.
  2. **App-launch splash** — single-line "Powered by…" credit visible ≤ 2s at boot. Implementation: HTML splash in `apps/desktop/index.html` for fastest paint (React replaces it on `createRoot()`).
  3. **Settings → About panel** — new `apps/desktop/src/views/SettingsView.tsx` mirroring `MemoryView.tsx`'s shell/HUDFrame pattern. Renders the full attribution list with per-component version strings + a link to LICENSES.md. Routed from the existing `settings` rail item in `App.tsx`.
- A **version-strings table** added alongside `attribution.ts` (preferred: new `packages/core/src/versions.ts`) that mirrors the Tier-1 pins in `docs/pinned-versions.md`. Hand-maintained for v1; future story can mechanize the scrape.
- A **`renderAttributionMarkdown()`** helper added to `attribution.ts` (or sibling) that produces a structured object the UI can render with React (not just plain text). The existing `renderAttributionText()` stays as-is for the splash + Settings clipboard-copy use cases.
- Tests for the new helpers + a guardrail that every BUNDLED entry has `name`, `license`, `source` set.
- Sprint-status row flipped to ✅ Done; story frontmatter records `code_review_a` verdict.

**IS NOT:**

- A creation of `attribution.ts` from scratch. It already exists (12 BUNDLED + 5 INTEGRATED entries + `renderAttributionText()`).
- A full Settings view with Personas / Vault / Theme / Updates sub-panels. Story 1.19 ships **About** as the first/only sub-panel; future stories can add more.
- A live scrape of `pinned-versions.md` at build time. Hand-maintained `versions.ts` for v1; the discipline of updating it when `pinned-versions.md` changes is captured as a note inside `versions.ts`.
- A native Tauri splash window. The in-document HTML splash satisfies "≤ 2 seconds at startup" without the multi-window complexity.
- An update to `LICENSES.md` content. That's Story 1.3's source-of-truth; we link to it, don't rewrite it.
- Changes to `attribution.ts`'s existing `BUNDLED` / `INTEGRATED` arrays unless a license/copyright field is actually wrong. (Audit is Story 1.3's responsibility.)

### Splash mechanism — HTML in `index.html`

The cleanest pattern: include the splash markup directly inside `<div id="root">` in `index.html`. Browser paints it on first frame (before any JS runs). When `main.tsx` calls `createRoot(root).render(<App />)`, React replaces the `#root` children with the real shell. The splash is automatically gone — no timer-based "hide" logic.

```html
<div id="root">
  <div class="boot-splash">
    <img src="/monogram.png" alt="4neverCompany OS" class="boot-splash-logo" />
    <p class="boot-splash-credit">
      Powered by Paperclip · Hermes Agent · BMAD Method · Tauri · Zellij · Claude Code
    </p>
  </div>
</div>
```

The `.boot-splash` CSS lives inline in `<style>` inside `<head>` so the paint doesn't wait for a stylesheet GET. Once React mounts (~200–800ms depending on cold start), the splash vanishes naturally — comfortably under the AC's 2-second cap.

The short credit string is constructed at build time from `renderAttributionShort(BUNDLED)` and injected into `index.html` via a Vite plugin OR — simpler — hand-maintained to match the source. **Recommend: hand-maintained.** The set of bundled-upstream names changes infrequently (when we add or remove a dependency) and the tests guard against drift between the canonical source and what the splash shows.

## Risks

- **Drift between `attribution.ts` and `pinned-versions.md`** — when a Tier-1 component bumps in `pinned-versions.md`, someone has to update `versions.ts`. Mitigation: `attribution.test.ts` includes a guardrail that fails if any `BUNDLED` entry is missing from `VERSIONS`. Doesn't catch stale versions (no automated diff against `pinned-versions.md`); flag as "discipline note" in `versions.ts` module-doc + a `references/reference-bmad-method-workflow.md` addition.
- **DoneStep visual regression.** The wizard's `DoneStep.tsx` currently has a compact, centered help-copy line. Replacing it with a full attribution block changes the visual weight of the card. Mitigation: keep the card compact — show the top-3 bundled upstreams inline + "See full credits in Settings → About →" CTA. The full block lives in Settings → About.
- **Splash flash-of-unstyled-content.** Browser paints `#root`'s HTML splash on first frame, then React mounts and replaces it. If the React shell takes > 2s to render, the splash exceeds the AC's "≤ 2 seconds" cap. Mitigation: monitor React mount time during visual smoke; if slow, code-split the shell. Should not be a concern at 1.19 scope.
- **`tauri-plugin-opener` scope.** The "Read full LICENSES.md →" button needs to open the LICENSES.md file. If the opener plugin's scope (in `tauri.conf.json`'s permissions) doesn't allow opening repo-rooted files, the button silently fails. Mitigation: verify scope in the dev story; if needed, expand `dialog:allow-open` analog or add `opener:allow-open-url` for `file://` scheme.
- **Settings rail item conflicts with future polish.** Story 1.19 owns the entire `settings` rail item slot. Future stories that add Personas / Vault / Theme sub-panels under Settings will edit `SettingsView.tsx`; this story should leave a clear "TODO(M2): Personas/Vault sub-panels" comment so the extensibility story isn't surprising.
